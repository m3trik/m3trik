- **Runtime-vs-static API drift gate: `verify_runtime_surface.py` + `Check-RuntimeSurface.ps1` (2026-07-06).** The static registry walker (`generate_api_registry.py`) parses source with `ast` â€” it can never see a member that only exists at runtime (metaclass/mixin injection) or notice a decorator that shadows a public method, so a committed registry can silently over-promise a member a live class doesn't actually have. New `verify_runtime_surface.py` closes that gap by diffing two independent introspection producers of the same public *class surface*: the committed static registry vs. `pythontk.HelpMixin`'s live introspection (`_collect_records`, own-body members only, so inherited-mixin members are excluded symmetrically and never false-flag). Both producers now share one wire format â€” `SymbolRecord` (see below) â€” so the comparison needs no ad-hoc translation. Drift is classified by member NAME, deliberately not signature string (the two engines render signatures differently): `missing` (registry promises a member the live class lacks â€” a real contract break or a stale registry) â†’ FAIL; `added` (live-only, e.g. metaclass injection) and `kind_changed` (e.g. a wrapping decorator turns a `@staticmethod` into a plain `method` at runtime) are advisory notes only. Two execution modes, because the DCC packages can't be imported on cloud CI: `verify <pkg>` imports in-process (pythontk; uitk under offscreen Qt); mayatk/blendertk instead `dump` their live surface from a fresh headless session (mayapy / `blender --background`) to a gitignored `API_RUNTIME.json` artifact, then `verify --runtime <artifact>` diffs it out-of-process. `Check-RuntimeSurface.ps1` orchestrates all four packages (skipping any runtime not installed on the machine) and is wired into the weekly `ClaudeContextBudget` task via `Invoke-ContextBudgetCheck.ps1` â€” its own log (`%LOCALAPPDATA%\claude-runtime-surface.log`) and a toast on a `missing`-member FAIL. `check_context_budget.py` grew a matching opt-in pythontk spot-check (`check_runtime_surface`, gated behind the new `--no-runtime` flag) for a convenient local run; cloud CI (`refresh-api-registry.yml`) now passes `--no-runtime` explicitly to stay import-free, which is the whole point of keeping the registry AST-only. Documented in a new "Runtime API drift" section of `docs/CONTEXT_BUDGET.md`.

- **`generate_api_registry.py` â€” shared `SymbolRecord` DTO + property-accessor phantom-member fix (2026-07-06).** The generator's private `SymbolEntry` dataclass is replaced by `SymbolRecord`, now the single shared shape for a public symbol across the two producers that need to agree on one: this generator's static `ast` walk, and `pythontk.HelpMixin`'s live `inspect` introspection powering the new runtime-vs-static drift gate (above). It lives at `pythontk/pythontk/core_utils/symbol_record.py` (zero non-stdlib imports) but is loaded via `importlib.util.spec_from_file_location` rather than `import pythontk` â€” `_load_symbol_record()` registers the module in `sys.modules` *before* `exec_module`, because a bare load leaves `@dataclass`'s annotation resolution looking up `sys.modules[cls.__module__]` and finding `None` â€” keeping the generator dependency-free and runnable on a bare CI box exactly as before. Markdown row rendering moved off the generator's private `_emit_member_line` and onto `SymbolRecord.to_registry_row()` (one renderer, shared by both producers). Separately: a property's `@x.setter`/`@x.deleter` was being walked and emitted as its own class member â€” double-listing the property and mislabeling the setter as a plain `method` (`.setter`/`.deleter` isn't a kind `_decorator_kinds` recognizes). New `_is_property_accessor()` skips them during the class-member walk; the `@property` getter itself is still emitted exactly once, correctly kinded `property`. `docs/API_SHADOWS.md` regenerated as a byproduct of validating this change. - **`push.ps1` â€” no more "phantom publishes" when re-running a cascade over a package with no real changes (2026-07-01).** Two coupled defects. (1) Step 4 (Merge to Main) was gated only on the `-Merge` flag, so it ignored the `$needsMerge = $false` that step 2 sets â€” the `"Dev is ahead only due to dev bump (skipping merge)"` message was a no-op, and step 4 merged the bumped version to main anyway, tripping `publish.yml` into shipping an unintended release (observed: `pythontk 0.8.77` auto-published to PyPI on a cascade re-run with no pythontk changes), then **mis-tagged** because `$localStrictVersions` still held the previous version â€” leaving an *untagged* PyPI version that violates the publishâ†’tagâ†’release invariant. Step 4 now gates on `$Merge -and $needsMerge`. (2) But `$needsMerge` alone was unsafe: `Test-OnlyDevBumpChanges` listed `pyproject.toml`/`README.md` as "bump-only" files, so a **dependency-cascade release** (whose only change is the upstream pin in `pyproject.toml` + the version) would have been classed as skippable and its merge dropped â€” silently defeating the whole point of the cascade. Since all four strict packages use dynamic versioning (`version = {attr = "<pkg>.__version__"}`), a pure bump touches **only** `<pkg>/__init__.py`; the allow-list is narrowed to exactly that. Net: a version-only bump skips (no phantom); a pin/doc/code change keeps `$needsMerge = $true` and merges + propagates as before. (The one phantom `0.8.77` was retroactively tagged + released by hand to restore the invariant.)
- **`generate_api_registry.py` / `check_context_budget.py` â€” `--check` ignores the cosmetic generation date (2026-07-01).** The `_Generated: <date>_` markdown line (and the JSON `"generated_at"` field) made every *untouched* registry read as "stale (would be rewritten)" the moment the day rolled over â€” so a cascade on a new day recurringly failed the budget gate and forced a date-only re-commit of packages that hadn't changed. The staleness comparison now strips the generation-date line(s) before diffing (via `_strip_generated`, mirroring the existing `API_CHANGES.md` exclusion); the written output still carries the visible date, but a date-only difference is no longer a staleness signal. Real content changes are still caught (they differ in more than the date).
- **`generate_api_registry.py` â€” deterministic module ordering across platforms (2026-06-27).** `_iter_py_files` sorted `Path` objects, but `WindowsPath` compares case-insensitively while `PosixPath` (CI/Linux) compares case-sensitively â€” so a local (Windows) regen ordered same-prefix modules differently from CI (e.g. `table_actions.py` vs `tableWidget.py`), making committed registries drift between local and CI runs and recurringly tripping `check_context_budget.py --check` after a cascade. - **`push.ps1` â€” every publish now creates an annotated `vX.Y.Z` git tag + a GitHub Release (2026-06-27).** Closes the gap where the curated `CHANGELOG.md` lived only as a file on a branch, never attached to anything git-native. After the devâ†’main merge + successful publish workflow, `push.ps1` tags the released `origin/main` commit `vX.Y.Z` (idempotent â€” skips if the tag already exists) and, when this release added CHANGELOG entries, cuts a GitHub Release whose notes are exactly those additions. Notes come from `Get-ChangelogDelta` = the lines added to `CHANGELOG.md` on dev vs the last-released `origin/main` (`git diff origin/main..dev`) â€” a deterministic boundary needing **no** change to the existing dated-prose changelog (no `[Unreleased]` section, no commit-message convention). A release that added no CHANGELOG entries (docstring-only / dependency bump) is still tagged, just without a Release body. The tag/Release step is additive and **non-fatal** â€” it runs only after PyPI publish + main merge already succeeded, so a `gh`/tag hiccup can never abort a release (add the tag manually if so). `common.ps1` `Merge-ToMain` also now auto-resolves a `CHANGELOG.md` merge conflict by taking dev's copy (the post-release state is always dev's), matching the existing `__init__.py`/`pyproject.toml` handling. The stamp/`[Unreleased]` model was evaluated and rejected: the repo's changelogs are richly hand-curated dated prose, and a `git diff` boundary delivers the same git-native Release with zero disruption to that habit.
